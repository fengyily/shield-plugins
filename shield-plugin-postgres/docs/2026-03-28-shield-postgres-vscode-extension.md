# 我用 7MB 的 Go 二进制替代了 pgAdmin

做后端开发这些年，pgAdmin 和 DBeaver 一直是我管理 PostgreSQL 的默认选择。它们功能强大，但说实话——启动慢、界面臃肿、和我的日常开发工作流是割裂的。我大部分时间都在 VS Code 里写代码，却要频繁切出去操作数据库。

前段时间开始做 Shield CLI 的 PostgreSQL 插件，最初只是想解决自己的痒点：**能不能在终端或编辑器里，用最轻的方式连上数据库，做完该做的事？**

现在这个东西跑起来了，也做了 VS Code 扩展，写篇文章聊聊它的思路和一些技术选择。

## 它是什么

Shield PostgreSQL 是一个 Go 写的单文件二进制（约 7MB），启动后在本地起一个 Web 服务，提供浏览器端的数据库管理界面。它可以：

- 通过 Shield CLI 的插件协议运行
- 作为 VS Code 扩展直接使用
- 或者用 Docker 跑一个 9MB 的容器，当轻量版 pgAdmin 用

没有 Electron，没有 JVM，没有 node_modules 黑洞。前端是纯 JavaScript + SVG，全部嵌入 Go 二进制里。

## 日常够用的功能

做这个工具的原则是**够用就好**。日常操作 PostgreSQL 最频繁的事情，它都能做：

- **Schema 浏览**：树状展开 Schema → Table → Columns / Indexes
- **SQL 编辑器**：多标签页、语法高亮、`Cmd+Enter` 执行
- **行级编辑**：直接在表格里增删改行数据
- **表结构管理**：建表、改列、加索引，带实时 SQL 预览
- **CSV 导出**和列排序这些零碎功能

还有一个 Read-Only 模式，后端和前端双重拦截，给生产库连接时心里踏实一些。

## ER 图是意外收获

本来没打算做 ER 图，但在梳理一个老项目的表关系时，发现现有工具要么需要额外安装，要么生成的图是静态的、不能交互。于是顺手做了一个基于 SVG 的交互式 ER 图：

- 拖拽移动表的位置，滚轮缩放
- 从一个表的列拖到另一个表的列，直接创建外键关系（会校验类型是否匹配）
- 右键菜单管理表和列
- 支持 Grid、水平、垂直、放射状等多种自动布局
- 导出 SVG 或整个 Schema 的 SQL

做完之后发现，这个 ER 图反而成了用得最多的功能——理解陌生项目的数据库结构时特别有用。

## 多人协作：和同事一起看同一张图

有一次在排查一个跨服务的数据问题，需要和同事一起看数据库 Schema。屏幕共享总是不够直观，于是加了 WebSocket 实时协作：

- 多个人连同一个数据库时，能看到彼此的光标位置（带名字和颜色标识）
- 一个人拖动表的位置，其他人实时同步看到
- 有人改了表结构或外键，所有人自动刷新
- 可以"跟随"某个人的视角，他移到哪你就看到哪

这个功能在 Code Review 讨论数据模型设计、或者新人 Onboarding 熟悉系统时意外地好用。每个连接会分配一个随机动物名字和颜色，不需要登录。

## VS Code 扩展的集成思路

VS Code 扩展做的事情很简单：

1. 在侧边栏管理数据库连接（保存、编辑、测试）
2. 点击连接时，自动启动 Go 插件进程
3. 通过 stdin/stdout 的 JSON 协议通信，拿到 Web 端口
4. 在浏览器里打开管理界面

之所以没有把 UI 做成 VS Code 的 WebView Panel，是因为数据库管理界面需要比较大的操作空间，浏览器标签页的体验反而比嵌入式面板好。而且这样 Web UI 也可以独立使用，不绑定 VS Code。

插件协议本身很简洁——就是 JSON 行，一个 `start` 动作带上连接配置，插件返回 `ready` 和端口号。这个协议设计让 Shield CLI 可以用同样的方式管理不同数据库的插件（Redis 等后续会做）。

## 技术选择的取舍

几个有意识的决定：

**前端不用框架**。纯 JavaScript 操作 DOM 和 SVG。107KB 的 ER 图模块全手写，没有 D3 或 React。代价是代码量大一些，好处是零依赖、启动快、打包进二进制不需要构建流程。

**Go embed 嵌入静态文件**。`go:embed static/*` 把前端文件编译进二进制，分发时就是一个文件，不需要考虑静态资源路径问题。

**WebSocket 用 gorilla/websocket**。虽然 Go 标准库的 `net/http` 能处理 WebSocket 升级，但 gorilla 的 API 更成熟，处理 ping/pong 和连接管理更省心。

**不做用户系统**。协作功能用随机名字 + 颜色标识用户，不需要注册登录。面向的场景是团队内部使用，信任边界在网络层而不是应用层。

## 适合什么场景

这不是要替代所有数据库管理工具。更适合的场景是：

- 日常开发中快速查数据、跑 SQL、改表结构
- 理解陌生项目的数据库 Schema（ER 图）
- 团队一起讨论数据模型设计（协作模式）
- 在服务器上临时跑一个轻量管理界面（Docker 部署，9MB）
- 生产库的只读浏览（Read-Only 模式）

如果你需要复杂的查询计划分析、存储过程调试、或者企业级的权限管理，pgAdmin 和 DBeaver 仍然是更好的选择。
